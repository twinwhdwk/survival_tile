# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A real-time multiplayer "last tile standing" browser game (Spleef / Fall Guys Hex-A-Gone style) built on Phaser 3 + Socket.io. Players spawn on a grid of tiles that collapse when stepped on; survivors advance through a tournament bracket that culminates in boss fights.

## Commands

```
npm run build   # webpack build: server bundle then client bundle -> dist/server.bundle.js, public/js/bundle.js
npm start       # node dist/server.bundle.js (runs the already-built bundle; requires npm run build first)
npm run client  # webpack --watch for the client bundle only
npm run server  # webpack --watch for the server bundle only (uses nodemon-webpack-plugin to restart on change)
```

There is no test suite. `npm run lint` runs ESLint (added as a devDependency — the bare `.eslintrc.js` alone was previously unusable via a bare `npx eslint`, since that resolves to whatever ESLint happens to be cached globally, which may be a major version too new for this config's format).

**Build environment quirk**: `npm run build` runs webpack 4, whose hashing breaks under Node 17+'s default OpenSSL 3 provider. On a modern local Node install, prefix the build with `NODE_OPTIONS=--openssl-legacy-provider`. This is a **build-time-only** requirement — `npm start` (running the already-built bundle) does not need the flag.

`PORT` env var controls the listen port (defaults to an OS-assigned port if unset).

## Architecture

**Server-authoritative, single global Node process.** All game logic (tile state, timers, scoring, elimination) runs server-side in `src/server/`; clients in `src/client/` only render state and send input. There is no database — all state (lobby, rooms, tournament bracket) lives in memory in `server.js` and is lost on process restart. Because of this, any cloud deployment must be pinned to a single instance (no horizontal scaling), or player state fragments across replicas.

### Server (`src/server/`)

- **`server.js`** — global orchestration singleton. Owns `globalPhase` (`'LOBBY'` | `'TOURNAMENT'`), `lobbyPlayers`, a `rooms` Map (roomId -> `Room`), `socketRoomMap` (socketId -> roomId), the tournament bracket state (`currentLineages`, `currentStage`, `stagePending`, `stageResults`, `finalRankings`), and `adminSockets` (see Admin mode below). A single `setInterval(tickAllRooms, 1000)` drives every room's round-state checks; each room's tick and each stage's room creation are individually try/caught so one bad room can't stall or hang the rest of a large tournament.
- **`Room.js`** — one instance per match. Modes are `'SURVIVAL'` (stage 1: survive inside a closing rectangular boundary until the timer runs out — see "Notable behaviors" below, there is no separate rescue-platform tile or state) and `'BOSS'` (stage 2+: fixed team fights a boss with HP scaled to survivor count). Owns the tile map, collapse timers, elimination, boss damage/scoring, ghost-revive logic, and (SURVIVAL only) auto tile regeneration and bot movement. `finishRoom(reason)` is the single exit point for a match; it calls the `onFinished` callback *before* emitting `roomResult`, and bundles any returned tournament rankings into that same `roomResult` payload — this avoids a real race where a separate `tournamentEnded` broadcast could otherwise arrive before the client has finished transitioning off `GameScene`.
- **`src/shared/*Config.js`** — tuning constants imported by both client and server (map size, round timing, boss HP/scoring, room size, nickname length). These encode the agreed game rules; treat changes to them as game-design decisions, not routine tuning.

### Tournament bracket model

Stage 1 randomly chunks all lobby players into groups of `MAX_PLAYERS` (remainder folds into the last group rather than spawning an undersized group). Each later stage merges *adjacent* lineages (`mergeAdjacentLineages`: lineage 0+1, 2+3, ...) into one team that keeps its combined score and plays on together; an odd lineage out just carries forward alone. A lineage that loses every member is locked into `finalRankings` immediately and does not pass a score forward. This means group sizes and boss HP can legitimately differ between simultaneous stage-2+ rooms.

### Client (`src/client/`)

Four Phaser scenes in sequence: `LoginScene` -> `LobbyScene` -> `GameScene` -> `ResultScene` (which can loop back to `LobbyScene` or forward to the next `GameScene` for the next bracket stage on the same socket, without a page reload). `GameScene` handles both SURVIVAL and BOSS rendering, the boss HUD, ghost mode (translucent avatar + click-to-revive on collapsed tiles), and the mobile joystick. All of a scene's socket listeners are registered from a single `this.handlers` map and torn down symmetrically on the Phaser `shutdown` event — this matters because a stage-2 transition reuses the same socket and re-enters `GameScene` without a reload, so incomplete cleanup would double-register handlers.

Tile/particle visuals are procedurally generated at runtime (no image assets) via `src/client/utilities/EffectTextures.js` (`generateTileTextures`, `generateBackgroundTexture`) and `AnimalTextures.js` (`generateAnimalTextures`) — called once per scene's `create()` (both `GameScene` and `LoginScene` generate/reuse the same `bg_gradient` texture key), guarded internally against regenerating a texture key that already exists. `GameScene.textureForState()` maps a `TILE_STATE` to one of `tile_solid`/`tile_solid_b` (checkerboarded by `(row+col)%2`, purely cosmetic) or `tile_warning`; there is no separate "rescue" tile texture or state. Tile transitions (collapse, revive, warning pulse) are driven by tweens + particle emitters in `GameScene`, not plain color swaps — if you add a new tile state or visual, wire it through both `renderMap()` (initial paint) and `setTileState()` (live transitions), which intentionally use different rendering paths (initial paint vs. animated transition).

Beyond the four main scenes, `DashboardScene` gives a connected admin a multi-room summary grid for stage 1/2 (see "Admin mode" below) before handing off to a single spectated `GameScene` at stage 3+. `src/client/utilities/SceneFx.js` holds `createAmbientEmbers`, a purely-cosmetic effect shared identically across `LoginScene`/`LobbyScene`/`DashboardScene`/`ResultScene` — kept in one place specifically so a tweak to the shared "burning boundary" mood can't accidentally apply to some scenes and not others.

`src/client/net/socket.js` is a singleton connection with a disconnect/reconnect banner. Because Socket.io v2 reconnection assigns a **new** `socket.id`, the server can no longer recognize a reconnected client as the same player mid-match — the client deliberately forces a full `window.location.reload()` on reconnect rather than trying to resume in place, which resets cleanly to `LoginScene`.

### Socket protocol

Client emits: `join`, `startTournament`, `addBot`, `clearLobby`, `resetServer`, `playerMovement`, `reviveTile`, `adminCritical`, `adminShatterTiles`, `adminSpectateRoom`, `adminReturnToDashboard`.
Server emits: `lobbyUpdate`, `joinRejected`, `gameStarting`, `dashboardStarting`, `dashboardUpdate`, `tileWarning`, `tileCollapsed`, `tileRevived`, `reviveGaugeUpdate`, `lastStandActivated`, `playerRevived`, `massCollapseStarted`, `boundaryPulse`, `playerMoved`, `playerEliminated`, `bossDamaged`, `bossShatterSkill`, `roomResult`, `tournamentEnded`.
Keep these two lists in 1:1 correspondence when adding features — an earlier cleanup removed client handlers (`newPlayer`, `playerDisconnected`) that the server never actually emitted.

### Admin mode

Starting the tournament and adding test bots are gated behind an admin password (`ADMIN_PASSWORD` env var, defaults to `3927`), checked server-side only in the `join` handler and never shipped to the client. The password field on `LoginScene` is deliberately tucked into an inconspicuous corner element, not the main form. A matching password adds the socket to `adminSockets` (server.js); `broadcastLobby()` emits a *personalized* `lobbyUpdate` per connected socket (not a single `io.emit`) so each client can be told `isAdmin` for itself. `LobbyScene` hides the "게임 시작"/"봇 추가" buttons entirely for non-admins, but the server independently re-checks `adminSockets.has(socket.id)` in both the `startTournament` and `addBot` handlers — don't rely on the client-side hiding alone.

Bots added via `addBot` are synthetic `lobbyPlayers` entries (id `bot-N`, `isBot: true`) with no real socket. They flow through `chunkForInitialRound`/`Room` exactly like real players (`io.sockets.sockets[botId]` is simply `undefined`, which existing guards already handle), but have no client sending `playerMovement` — `Room.moveBotsRandomly()` drives them with a single-tile step through the same `movePlayerTo()` path a real player uses, so collapse/elimination behave identically. It's driven by its own `tickAllBots` interval in `server.js` (`BOT_MOVE_INTERVAL_MS`, currently 600ms — raised from an original 300ms for more human-like pacing, see "Notable behaviors" below) — deliberately *not* folded into the 1s `checkRoundState` tick, since a real player sends a steady stream of small movements every frame and bots stepping only once a second read as barely moving by comparison.

Bots have no socket, so nothing ever fires `disconnect` for them — if an admin adds bots and the tournament is never actually started, those `lobbyPlayers` entries sit there forever, silently inflating the count the *next* time someone starts a tournament (this actually happened: 2 real people ended up split into different rooms because stale bots from an earlier aborted test were still in the lobby). The admin-only "초기화" button (`clearLobby` event, `LobbyScene`) is a full reset, not just a stale-entry sweep: it removes every bot entry, and for every non-admin human entry it actively force-disconnects that socket (`liveSocket.disconnect(true)`) before dropping them from `lobbyPlayers` — so a currently-connected real player *is* kicked (their client runs the normal disconnect/reload flow and lands back on `LoginScene`), not just a disconnected one swept out of the roster. Any connected admin socket is explicitly skipped and stays seated. `clearLobby` only works in the `LOBBY` phase and only ever touches `lobbyPlayers`, not any in-progress `Room`.

"서버 초기화" (`resetServer` event, `server.js`'s `resetServerState()`) is a broader emergency reset for a stuck/misbehaving tournament: unlike `clearLobby` it works from *any* `globalPhase` and also tears down every room in `rooms` (each gets `room.finished = true` set directly — every one of `Room.js`'s own `setTimeout` callbacks already checks `this.finished` before doing anything, since `finishRoom()` sets it for exactly this reason, so this alone is enough to make any already-scheduled tile-warning/collapse/move-broadcast timer a silent no-op without needing to track down and cancel each one individually), then force-disconnects every non-admin socket the same way `clearLobby` already does. It's a *soft* reset only — the Node process and its event loop/timers keep running untouched (so this is ~instant, no Cloud Run cold-start downtime), and it does not recover from an actual process crash or memory corruption; the `uncaughtException`/`unhandledRejection` handlers already guard against those. Exposed as a plainly-visible dark-red button in `LobbyScene` (next to `clearLobby`'s), but only as a small, muted, corner-tucked button in `DashboardScene` — that screen's own comments already establish "nobody watching can tell an admin capability is here" (same reasoning as the C/S skills' unlabeled keys) since it may be projected on a TV for the whole event to see, and a big alarming reset button would break that.

Movement isn't just immediate-neighbor avoidance: `Room.findStepTowardSafety()` runs a shallow BFS (capped at `BOT_SEARCH_DEPTH`, 6 tiles) from the bot's current position, looking for the nearest `SOLID` tile that's also inside the current safe boundary, and returns only the first step of that route (the search reruns fresh every tick anyway, since the map keeps changing under it). Ties among multiple equally-near valid tiles are broken randomly — an earlier version of this returned the *first* match found, which always favored the same up/down/left/right check order and made bots visibly drift one direction instead of wandering. If the BFS can't find a path within its depth (e.g. sealed into an isolated pocket), `moveBotsRandomly()` falls back to its simpler immediate-neighbor logic: prefer a `SOLID` neighbor over one that's `WARNING`/`GONE` when any such option exists.

Both the BFS and the fallback also check `this.pendingTiles` explicitly, not just `tileMap` state — `triggerTileCollapse()` adds a tile to `pendingTiles` the instant it's scheduled, but the tile's *state* doesn't flip from `SOLID` to `WARNING` until `WARNING_DELAY_MS` (600ms) later. A tile that's already pending but still reads `SOLID` used to look like a perfectly safe target; bots would walk onto it and get caught by a fuse someone else had already lit. Found via a 36-bot load test (see below) and confirmed with server-side position/elimination tracing — worth remembering if you touch tile-safety logic anywhere else, since `tileMap` state alone is not a complete picture of what's actually about to disappear.

`Room.hasHumans` (computed once at construction) makes `eliminatePlayer`'s all-eliminated check also finish the room once every *non-bot* player is eliminated/disconnected, even if a bot is still wandering — otherwise a room a real player abandoned mid-test would linger with just a bot in it for the rest of the round, stalling `stagePending`. A room that was all-bots from the start (`hasHumans === false`, possible when humans land in other rooms of a larger admin test) is unaffected and plays out normally until every bot is actually eliminated.

Once several simultaneous rooms exist (stage 1/2), a connected admin doesn't get seated into any single one — `startStage()` instead sends them a one-time `dashboardStarting` plus periodic `dashboardUpdate` broadcasts (`DashboardScene`, fed by `Room.getSummary()` off the same 1s tick as everything else), a grid of per-room alive-count/score/timer/boss-HP cards. At stage 3+, once the bracket has narrowed to where a single full room is more useful than a summary grid, the admin instead gets joined into room index 0's channel as a spectator (`gameStarting` with `isSpectator: true`), rendered by the same `GameScene` real players use.

Hidden, admin-only "balance levers" exist per BOSS room: pressing "C" (armed via `Room.armCriticalHit()`) makes the *next* boss hit in a targeted room deal `CRITICAL_DAMAGE_MULTIPLIER`x damage, and "S" (`Room.triggerBossShatterSkill()`) instantly collapses `ADMIN_SHATTER_TILE_COUNT` random standing tiles through the normal `triggerTileCollapse()` path. Both are rate-limited per room (`ADMIN_CRITICAL_COOLDOWN_MS`/`ADMIN_SHATTER_COOLDOWN_MS`) and deliberately produce no visible cue beyond what normal play already looks like (a bigger-than-usual damage number; a `bossShatterSkill` broadcast that's just a camera shake/rumble, framed as "the boss did something," not as an admin intervention) — the admin's screen may be projected on a TV, and the whole point is nobody watching can tell it happened. In `DashboardScene` the C/S keys apply to whichever room card is currently selected (click to select); in a spectated `GameScene` (stage 3+) they apply to the one room being watched, since there's no grid to pick from there.

### Pre-round countdown

Every round (SURVIVAL and BOSS) starts with a `START_COUNTDOWN_MS` (10s) freeze: `Room.movePlayerTo()` silently no-ops for anyone — human or bot — while `Date.now() - this.roundStartTime < START_COUNTDOWN_MS`, since `moveBotsRandomly()` and real `playerMovement` events both funnel through it. `GameScene`'s countdown overlay (`countdownActive` flag, gates local input the same way in `update()`) is purely cosmetic on its own — the actual freeze is enforced server-side, so it can't be bypassed by a modified client. Note this eats into `BOUNDARY_SHRINK_GRACE_MS` (20s) from the same `roundStartTime` reference, so SURVIVAL's actual free-roam window is only ~10s (20s grace minus the 10s countdown), not the full 20s — that's presumably intentional, but worth knowing if round pacing ever feels off.

### Notable behaviors baked into the current design (not bugs)

- Boss-mode rounds never end in a loss on timeout — surviving players' score carries forward into the next stage. Only a full team wipeout (`all-eliminated`) knocks a lineage out of the tournament.
- Round length is fixed (`ROUND_DURATION_MS`); reaching the center early does not end a SURVIVAL round early, it just determines who's still standing when the timer expires or the boundary forces the issue.
- SURVIVAL rounds are a battle-royale-style closing boundary, not a one-time "mass collapse" event. For the first `BOUNDARY_SHRINK_GRACE_MS` (20s) the whole map is free-roam. After the grace period, `Room.shrinkBoundary()` insets all 4 edges inward by one ring at a time, every `BOUNDARY_SHRINK_INTERVAL_MS` (15s). The safe area is a shrinking *rectangle* (`Room.safeInset`, clamped independently per axis against `MAX_ROW_INSET`/`MAX_COL_INSET`), not a radius from center — the map is wider than tall, so a naive square/radius model burns only the two long edges on the first step while doing nothing to top/bottom, which let a player spawned on that edge get caught almost immediately after the grace period ended. Insetting each axis in lockstep instead peels a thin ring off *all four sides* every step, which is both fairer and reads better as "the boundary is closing in." There's no distinct "safe zone" tile state or color — tiles stay plain `SOLID` the whole time, and `Room.isSafeTile()`/`safeInset` alone decide what's still inside the shrinking boundary. The client instead draws the safe area as a glowing rectangle outline (`GameScene.updateBoundaryOutline()`), dimming everything outside it, built from the `safeBounds` field sent on `massCollapseStarted` (once, with the banner/big shake) and `boundaryPulse` (every ring after that, with a lighter shake only — no repeated banner spam). None of this applies to BOSS rounds.
- `Room.autoRegenerateTiles()` runs for the *entire* SURVIVAL round now, not just the pre-boundary grace period, and is threshold-triggered rather than fixed-schedule: once fewer than `AUTO_REGEN_SOLID_RATIO_THRESHOLD` (0.75) of the *current safe zone's* tiles are still `SOLID`, it restores a burst sized `AUTO_REGEN_BASE_BURST + AUTO_REGEN_BURST_PER_ALIVE_PLAYER * aliveCount` (currently 5 + 2 per currently-alive player, so the burst scales with how many players are actually still consuming tiles right now instead of a flat number), rate-limited to at most once per `AUTO_REGEN_MIN_INTERVAL_MS` (1000ms). Candidates are restricted to `getSafeZoneTiles()` — never regenerate ground outside the shrinking boundary, that would defeat the point of it. This replaced an earlier fixed 9-tiles-per-3s design that turned itself off once the boundary activated, which a 36-participant (9-room) load test showed causing a **100% full-team-wipeout rate**, well before any round could resolve naturally. (The threshold itself was also raised from an original 0.5 to 0.75 — see `roundConfig.js`'s comment for why 0.5 in practice read as "nothing happens until the room is already in serious trouble.")
  - That load test (`gcloud`-deployed admin + 35 bots via `addBot`, driven by a standalone `socket.io-client` script, not Playwright — bots need no browser at all) is also what surfaced the `pendingTiles` bug documented above. Both fixes are real improvements, but **neither fully eliminates rapid wipeouts under sustained bot pressure** — traced with temporary server-side logging (position + elimination tracking, since removed) down to bots simply dying individually over time even with regen actively firing and the map nowhere near exhausted. The likely explanation: `MAX_PLAYERS` (4) bots each triggering a new tile collapse roughly every `BOT_MOVE_INTERVAL_MS` (300ms) is a *much* higher sustained consumption rate than real human play (which naturally pauses, hesitates, chats) — this may simply be an inherent limit of stress-testing with nonstop-moving bots rather than a bug to keep chasing. If you revisit this, don't assume it's still an open bug without re-testing against real/semi-realistic movement patterns first.
- `Room.js` and `server.js` are both defensively validated against malformed socket payloads (`Number.isFinite`/`Number.isInteger` checks) and wrapped with `process.on('uncaughtException'/'unhandledRejection', ...)` — this app is expected to run unattended for an entire multi-round event, so a single bad message or edge case should never take the whole server down.

## Deployment

Deploys as a container (`Dockerfile` in repo root, multi-stage: builds on `node:16-alpine` to avoid the OpenSSL/webpack4 issue, runs on `node:20-alpine`). Because all state is in-memory and single-process, any deploy target must be constrained to exactly one instance (e.g. Cloud Run with `--min-instances=1 --max-instances=1`).

`phaser` and `socket.io-client` are client-only (bundled by webpack into `public/js/bundle.js`, never imported under `src/server/`) and live in `devDependencies` for that reason — keep it that way, or the runtime image's `npm ci --omit=dev` will pull them in for nothing.

Cloud Run specifics already handled in `server.js`: `app.set('trust proxy', 1)` (Cloud Run terminates TLS at its edge), a lightweight `GET /api/health` (200 normally, 503 once shutdown has begun, so Cloud Run stops routing to a draining instance), and a `SIGTERM`/`SIGINT` handler that stops the game tick and calls `io.close()` (which — verified against socket.io v2's source — also closes the wrapped `http.Server`, so the shutdown callback only fires once the port is truly released) before `process.exit(0)`, with an 8s force-exit fallback. No custom "server restarting" event is sent to clients — the existing disconnect/reconnect-reload flow in `net/socket.js` already covers it.

**Dependency security**: run `npm audit --omit=dev` (not plain `npm audit`) to see what's actually reachable in the deployed image — the dev toolchain (webpack4/babel-era) carries the large majority of reported vulnerabilities but never ships. A plain `npm audit fix` (no `--force`) resolved every critical/high-severity finding in production deps without any breaking change. What's left after that is low/moderate-severity and requires a socket.io v2→v4 major bump to fully clear — don't do that casually; this codebase relies on v2-specific API behavior throughout (e.g. `io.sockets.sockets` being a plain object, not a Map) that changed in later majors.

**Do not name this health route `/healthz`.** Verified directly against a live deploy on this project's `*.run.app` domain: every other path (including deliberately-unmatched ones, which correctly reach Express's own 404 with our Helmet headers and `x-cloud-trace-context` intact) reaches the container fine, but `/healthz` specifically returned Google's own branded 404 page with none of those headers — i.e. it never reached the app at all. Something in Google's edge/GFE layer in front of `*.run.app` appears to intercept or cache that exact path. `/api/health` was picked as a working replacement.

First deploy to a brand-new GCP project will also hit IAM gaps that `gcloud run deploy --source` doesn't auto-resolve: the default Compute Engine service account (`<PROJECT_NUMBER>-compute@developer.gserviceaccount.com`, used as the Cloud Build service account here) needs `roles/storage.objectViewer` (to read its own uploaded source zip), `roles/logging.logWriter` (Cloud Build fails the whole build if it can't persist logs — happens even though the actual *Docker build* step itself shows SUCCESS), and `roles/artifactregistry.writer` (to push the built image). All three showed the *Docker build step* succeeding while the overall build still failed, which was the tell that this was a permissions issue, not a build issue.

**Short URL (`firebase.json`/`.firebaserc`)**: Firebase Hosting is used purely to alias the long `*.run.app` URL to a short, memorable `https://survival-tile-game.web.app` — the Hosting site (`survival-tile-game`) lives in this same GCP project (`project-ad2bbfa9-3823-4213-b0e`), a separate one from the user's older pre-existing "survival-tile" Firebase project (which is unrelated, on a different GCP project, and only ever had Realtime Database, not Hosting). `firebase.json` uses a plain HTTP **redirect** to the Cloud Run URL, not a Hosting `rewrites: { run: {...} }` proxy — verified directly (a `socket.io-client` transport check) that Firebase Hosting's Cloud Run rewrite integration does not pass WebSocket upgrades through, silently downgrading every connection to HTTP long-polling (still functionally correct, but adds latency and per-movement request overhead that would hurt this game's frequent-small-packet traffic, especially with several players moving at once). A redirect avoids that entirely: the browser is sent to the real Cloud Run origin on the very first request, so every actual game connection (including the WebSocket upgrade) happens directly against Cloud Run with no degradation — the short URL is purely an entry point, never a proxy in the traffic path.
